## ⭐ 한눈에 보기 (3-Line Summary)

1. **무엇을** — 해외(미국) K-Beauty의 SNS 트렌드 · 리뷰(VOC) · 인기 제품을 AI가 분석해 **"글로벌 고객이 기대하는 포인트 + 그 근거"** 를 도출
2. **어떻게** — Claude 기반 **Agentic 파이프라인**(수집 → 통계·LLM 분석 → 의사결정 → 리포트)을 **매주 무인 자동 실행**
3. **결과물** — AP 브랜드가 바로 쓰는 **3-탭 인터랙티브 리포트** (📈 트렌드 / 🛍️ 제품 전략 / ✈️ **방한 추천**)

> 🏆 **차별점** — 단순히 "결과"만 던지는 블랙박스가 아니라, **모든 판단 기준(수식·임계값)을 직접 설계해 공개**하는 화이트박스. "왜 이 인사이트인가"의 근거를 함께 제공해 **AP의 후속 기획·전략·개발을 독려**한다.

## 1. 배경 & 문제 정의

### 📊 시장 배경 — 왜 지금, 왜 미국인가
K-Beauty는 미국에서 폭발적으로 성장하며, **AP의 글로벌 무게중심도 미국으로 이동** 중이다.

- 2025년 미국 K-뷰티 매출 **약 $20억 (+37% YoY)**, 한국은 프랑스를 제치고 **對美 화장품 수출 1위**
- 2025년 AP 미주 매출 **+20%** → **미국 매출이 처음으로 중국을 추월** (코스알엑스 인수 효과로 미국 매출 2,867억→5,246억)
- TikTok 발(發) K-뷰티 붐이 Sephora·Ulta 실매출로 전환, **올리브영 LA 매장 2026년 오픈 예정**

### 🎯 문제 — "포인트"만으로는 부족하다
> 주제는 "글로벌 고객이 기대하는 포인트"를 알려달라는 것. 그러나 **포인트만으로는 실제 기획·전략·개발로 이어지지 않는다.**

- AP가 그 포인트로 **실행**하려면, "왜 이 포인트인가"의 **구체적 근거·신뢰성**이 함께 와야 한다.
- 게다가 이 문제는 **정답(ground-truth) 라벨이 없다** — "이 트렌드가 진짜 기회였는가"를 사전에 알 수 없다.
- → 그래서 우리는 **정확도 최적화 대신, 모든 기준을 투명하게 설계·공개하는 화이트박스** 방식을 택했다.

## 2. 데이터 — 소개 & 전처리

### 🗂️ 데이터 소스 (왜 이 플랫폼인가)

| 구분 | 플랫폼 | 선정 이유 |
|------|--------|-----------|
| 리테일 | **OliveYoung Global** | 방한 외국인 **10명 중 8명이 방문**(2025년 1~5월 720만 중 596만 구매), 화장품 결제 88%·구매액 1조원. **글로벌+한국 랭킹 동시 제공** |
| 리테일 | **Sephora USA** | 미국 주류 리테일 침투도 + 리뷰(BazaarVoice) |
| SNS | **TikTok** | K-Beauty 트렌드 **최초 발화 채널** (신규 성분·제형이 먼저 등장) |
| SNS | **YouTube** | 리뷰·하울·루틴 등 **심층 콘텐츠** |

> **TikTok(선행) → YouTube(확산)** 교차 검증으로 "반짝 유행 vs 진짜 트렌드"를 구분.

### ⚙️ 전처리에서 풀어야 했던 2가지 난제

**난제 ①: SNS 노이즈** — 해시태그만 믿으면 "생일파티"·"100k뷰" 같은 비관련 포스트가 섞임
→ **캡션 텍스트를 직접 분석**해 K-Beauty 성분/제형 키워드 사전에 없으면 제거 (2단계 필터)

**난제 ②: 같은 제품, 다른 이름** — 동일 제품도 플랫폼마다 표기가 다름
```
OliveYoung : Advanced Snail 96 Mucin Power Essence
Sephora    : Snail Mucin 96% Power Repairing Essence 100mL   ← 같은 제품!
```
→ **정규화(마케팅 수식어·용량·관사 제거) + 퍼지 매칭(rapidfuzz, 임계값 85)** 으로 동일 product_id 통합
→ 통합 못 하면 OY 인기 신호와 Sephora 리뷰가 흩어져 **분석이 깨짐** → 정합성의 핵심 기반

## 3. 모델링 & 분석 — 시스템 아키텍처

```
Orchestrator (Claude API Tool Use — DB 상태 보고 필요한 단계만 실행)
   ├── CollectionAgent : SNS·리테일·리뷰 수집 + 노이즈 필터
   ├── AnalysisAgent   : 통계(Z-score·모멘텀) + Claude LLM 인사이트
   └── DecisionAgent   : 3가지 실행형 인사이트 생성
            ├── TrendInsightBuilder    (기회 유형 × 트렌드 모양)
            ├── ProductInsightBuilder  (VOC 모멘텀 중심)
            └── InboundPickBuilder     (방한 추천)
```

- **통계** : Z-score(순간 급등) + 모멘텀(3주 지속 상승) + 크로스플랫폼 검증
- **LLM (Claude Opus 4.8)** : 급등 키워드 → AP 제품 연결 / 경쟁사 탐지 / 소비자 니즈 요약 (+ SNS 캡션 근거 인용)
- **Agentic** : 고정 순서가 아니라 Claude가 DB 상태를 보고 **필요한 단계만 판단·실행** (ReAct 루프)

## 4. 💡 핵심 의사결정 — Before → After → Why

> 정답 라벨이 없는 문제이므로, **"왜 이렇게 설계했는가"의 논리가 곧 실력**이다. 프로젝트 진행 중 다음 3가지를 직접 재설계했다.

### 결정 ① 트렌드 분류를 "1축 → 2축"으로 분리
| | |
|---|---|
| **Before** | 단일 점수로 트렌드를 줄세움 |
| **After** | **기회 유형**(누구의 기회) × **트렌드 모양**(급등/지속) 직교 2축 |
| **Why** | 단일 점수는 "누구의 기회인지"와 "얼마나 지속될지"를 뒤섞음. 2축이면 *"essence = AP 기회 × 지속 상승 → 중장기 투자"* vs *"vitamin_c = 차별화 × 순간 급등 → 짧고 빠르게"* 처럼 **액션이 달라지는 해석** 가능 |

### 결정 ② 제품 전략을 "리테일 순위 → VOC 모멘텀" 중심으로
| | |
|---|---|
| **Before** | 리테일 순위 70% 비중 |
| **After** | **VOC 모멘텀(리뷰 유입 속도 + 별점 추세)** 주 신호, 리테일은 보조 |
| **Why** | 리테일 순위는 **단일 플랫폼(OY) 의존·결측 多·후행 지표**. 반면 리뷰는 **선행 신호**. 특히 "수집 샘플 수"가 아닌 **"플랫폼 전체 리뷰 수의 주간 증감"** 을 써서 *수집기 한계가 아닌 실제 소비자 활동*을 측정 |

### 결정 ③ 방한 추천 가중치 재조정
| | |
|---|---|
| **Before** | 순위 70% + SNS·VOC 22% |
| **After** | **순위 35% + SNS·VOC 65%** |
| **Why** | 방한 추천의 본질은 *"외국인이 한국 와서 살 가치"* → **"지금 해외에서 화제(SNS) + 실제 만족(VOC)"** 이 핵심. SNS도 단순 연결 여부(0/1)가 아니라 **트렌드 강도(z·모멘텀 정규화)** 로 반영 |

> 🔑 **이 3가지 결정의 공통 원리**: 단일·정적·후행 지표 의존을 줄이고, **동적·선행 신호 + 투명한 수식**으로 전환. 모든 점수는 리포트 하단에 수식으로 공개되어 *"왜 이 제품이 PUSH_NOW인가"* 를 추적 가능.

## 5. 성과 & 결과

> 정답 라벨이 없으므로 "정확도"가 아닌 **① 도출 인사이트 ② 설계 타당성 ③ 처리 커버리지** 로 제시.

| 관점 | 내용 |
|------|------|
| **도출 인사이트** | 주당 SNS 키워드 ~88개 → 통계·노이즈 필터로 핵심 트렌드 선별 · AP 제품 100여 개 VOC 4분면 분류 · 방한 추천 Top 15 |
| **설계 타당성** | 모든 점수가 **공개 수식**으로 산출 → 재현·검증 가능 · 신호 간 **상호 검증**(순간↔지속, TikTok↔YouTube, SNS↔VOC) |
| **처리 커버리지** | **5개 플랫폼** 자동 수집 · AP **13개 브랜드 / 721개 제품** 카탈로그 · **매주 무인 자동 실행** |

**실제 검증 사례** — 방한 추천에서 SNS 트렌드 강도가 높은 제품이, VOC만 높은(SNS 무연결) 제품을 앞섬 → *"지금 해외에서 화제인 제품"이 상위로* 올라오는 설계 의도대로 작동.

## 6. 자동화 — "지속적 인사이트 제공"

주제가 요구한 **"지속적으로 인사이트·이슈 알림"** 을 2층위로 구현.

- **Agentic Orchestrator** : Claude가 DB 상태를 보고 이미 처리된 주차는 자동 스킵 (불필요한 재수집·비용 방지)
- **정기 자동 실행** : 매주 목요일 전체 파이프라인 / 매월 1일 카탈로그 재구축 (스케줄러 + 실행 로그)

## 7. 🙋 나의 역할 & 배운 점

> *(아래는 예시 — 실제 본인 기여에 맞게 수정하세요)*

### 담당 / 기여
- 전체 멀티 에이전트 **아키텍처 설계 및 구현**
- 5개 플랫폼 **수집기 + 교차 플랫폼 제품 동일화** 파이프라인
- **3대 인사이트 산식 설계** (트렌드 2축 / 제품 VOC 모멘텀 / 방한픽 가중치)
- Claude **Agentic 파이프라인** + 인터랙티브 HTML 리포트

### 어려웠던 점 & 해결
| 문제 | 해결 |
|------|------|
| 플랫폼마다 제품명이 달라 동일 제품 연결 실패 | 정규화 + 퍼지 매칭(임계값 85)으로 통합 |
| VOC 수집이 일시적으로 실패해 결측 발생 | 제품별 재시도 + 직전 주차 폴백(출처 투명 표기) |
| 정답 라벨이 없어 "정확도"로 평가 불가 | 대리 신호 + 공개 수식으로 **타당성 기반 설계**로 전환 |

### 배운 점
- **정답이 없는 문제일수록, 결과보다 "판단 근거의 투명성"이 신뢰를 만든다.**
- 비즈니스 맥락(방한 동선·AP 미국 전략)을 데이터 설계에 직접 반영하는 경험.
